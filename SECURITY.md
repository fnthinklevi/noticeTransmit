# 安全政策（SECURITY）

[English](SECURITY-en.md) / 中文

本文档说明通知推送助手（noticeTransmit）的安全支持范围、漏洞报告方式与安全部署要求。

---

## 1. 支持版本

- **仅最新正式版本**接收安全修复。当前受支持版本：**v1.5.38（build 72）**。
- 旧版本（含已停止维护的版本）不再提供安全更新，请尽快升级到最新版。

---

## 2. 如何报告漏洞

> ⚠️ **请勿在公开 Issue 中披露漏洞利用细节或 PoC**，以免被滥用。请通过以下**私密渠道**报告。

**报告渠道（任选其一）：**
- GitHub 仓库的 **Security → Report a vulnerability**（私密安全公告，推荐）；
- 或私信维护者：**（请维护者在此填写邮箱 / 联系方式）**。

**报告中请尽量包含：**
- 漏洞类型与所在位置（文件路径 / 接口 / 模块）；
- 复现步骤（含环境、触发条件）；
- 影响范围与严重程度评估；
- 如有可能，附上修复建议。

**响应约定：**
- 目标在 **72 小时**内给出首次响应；
- 确认后协商合理的公开披露时间，遵循「负责任披露」原则。

## 3. 安全设计概览（范围）

- **通知内容仅本地处理与推送**：所有系统通知、短信、通讯录等仅在设备本地解析与转发，**不上传到任何第三方服务器**（除非用户自行配置的 Webhook 目标）。仅 Bugly 采集最小必要的崩溃统计（堆栈、设备型号、系统版本、应用版本）。
- **Webhook 由用户自配**：推送地址为用户自行填写的 URL，服务端不存储任何推送内容。
- **管理后台鉴权**：登录 Token 使用 bcrypt 哈希；支持 TOTP 二步验证（兼容 Google Authenticator）；TOTP secret 使用 **AES-256-GCM** 加密存储（密钥来自 `ENCRYPTION_KEY`）。
- **Token 传输**：仅接受 HTTP Header 传递，禁止在 URL 参数中携带。
- **防暴力破解**：10 分钟内验证码错误 3 次自动封锁 IP 240 小时；提供 8 个恢复码用于设备丢失找回。
- **安全随机数**：会话 ID 使用 `crypto.randomUUID()` 生成。

## 4. 安全部署与运维要求

### 4.1 密钥管理（重要）

`server/.env` 存放真实密钥（`ADMIN_TOKEN_HASH` 与 `ENCRYPTION_KEY`），**请勿将其提交到仓库**（已在 `.gitignore` 中忽略，仓库仅保留 `.env.example` 占位模板）。

部署规范：

1. 真实 `.env` **不纳入版本控制**，仅本地存在，切勿提交；
2. 通过环境变量注入密钥，**切勿硬编码**到代码或配置中；
3. 建议定期轮换密钥，降低长期暴露风险。

### 4.2 加密与传输

- `ENCRYPTION_KEY` 必须为 **64 位十六进制**（AES-256-GCM 需要 32 字节）。格式不合法时会被忽略，TOTP secret 将以**明文**存储，请提交前自检。
- CORS 默认**仅放行无 `Origin` 的原生请求**；Web 后台跨域访问需显式配置 `ALLOWED_ORIGINS` 白名单，**不要设为 `*`**。
- 经反向代理（如 Nginx）部署时，通过 `TRUST_PROXY` 配置代理跳数；配置不当会放大 IP 伪造、`X-Forwarded-For` 绕过限速等风险。
- 管理接口（`/api/admin/*`）务必置于可信网络，或叠加额外访问控制（如只监听内网、加 WAF / 口令网关）。

### 4.3 客户端

- 发布构建默认关闭 R8 混淆与资源压缩（见 `proguard-rules.pro`，可按需开启）；若开启请同步校验反混淆映射的安全保管。
- 通知监听、电池白名单、自启动等系统权限由用户授予；权限缺失属于功能受限，不视为安全漏洞。

---

## 5. 不在范围内（Out of Scope）

- 用户自行配置的第三方 Webhook 目标的服务端安全性。
- 设备已被 root / 提权后的本地数据保护。
- 因未授予必要系统权限导致的功能失效（非安全漏洞）。
- 依赖项上游漏洞：请通过对应上游渠道报告，并在本项目同步升级。
