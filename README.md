# 通知推送助手

监听 Android 通知栏消息，通过 Webhook 推送到指定渠道（企业微信、钉钉、飞书等）。

## 功能特性

- 🔔 **通知监听**：监听系统所有应用的通知栏消息
- 📱 **多平台支持**：微信、QQ、短信、来电、系统通知等
- 🔗 **Webhook 多通道**：支持同时配置多个 Webhook URL
- 🔋 **电量提醒**：充电、充满、低电量（30%/20%）提醒
- 📋 **历史记录**：本地保存通知推送历史，支持搜索和导出
- 📱 **应用筛选**：自定义选择哪些应用的通知需要推送
- 🏷️ **关键词过滤**：支持白名单和黑名单关键词过滤
- 🛡️ **后台保活**：前台服务 + 电量优化白名单 + 开机自启动
- 📤 **多格式适配**：自动适配企业微信、钉钉、飞书等平台格式

### v1.5.5 (build 38) - 2026-06-30

**ANR 修复：应用无响应问题**
- 修复更新后首次打开应用出现"软件无反应，是否结束运行"的问题
- 根本原因：NotificationMonitorService.onCreate() 在主线程执行耗时操作（加载配置、读取热更新文件、设备名文件I/O）
- 解决方案：将 loadConfigFromPrefs() 和 loadHotfixConfig() 移到 IO 线程异步执行
- 优化服务启动流程：先启动前台服务和注册广播，再后台加载配置
- 确保服务快速响应，不阻塞主线程

### v1.5.4 (build 37) - 2026-06-30

**设备名持久化彻底修复**
- 修复更多页设备名在软件更新后仍重置为默认值的问题
- 根本原因：Flutter端_getDeviceInfo()与_loadAllSettings()存在竞态条件，前者可能先生成默认值并写入文件，覆盖文件中已有的正确值
- 解决方案：统一设备名初始化入口，全部交由原生端getDeviceName处理
- 移除Flutter端_getDeviceInfo()中的默认设备名生成逻辑
- 原生端getDeviceName在文件和SharedPreferences都为空时，自动生成默认设备名并保存
- Flutter端_loadAllSettings优先从原生端获取设备名，失败时才回退到SharedPreferences

## 技术架构

- **前端**：Flutter (Dart)
- **原生服务**：Kotlin (Android)
- **通知监听**：NotificationListenerService
- **后台保活**：Foreground Service + WakeLock + WifiLock
- **跨端通信**：MethodChannel

## 权限说明

- `BIND_NOTIFICATION_LISTENER_SERVICE` — 监听系统通知
- `FOREGROUND_SERVICE` — 前台服务保活
- `RECEIVE_BOOT_COMPLETED` — 开机自启动
- `READ_SMS` / `READ_PHONE_STATE` — 短信/来电增强识别
- `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` — 电量优化白名单

## 签名配置

使用自定义 JKS 密钥进行 release 签名，配置文件位于 `android/key.properties`。
